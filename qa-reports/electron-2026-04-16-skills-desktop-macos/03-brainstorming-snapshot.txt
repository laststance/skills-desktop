  - cell "## Anti-Pattern: \"This Is Too Simple To Need A Design\"" [ref=e72]
  - cell "17" [ref=e73]
  - cell "  " [ref=e74]
  - cell "18" [ref=e75]
  - cell "Every project goes through this process. A todo list, a single-function utility, a config change — all of them. \"Simple\" projects are where unexamined assumptions cause the most wasted work. The design can be short (a few sentences for truly simple projects), but you MUST present it and get approval." [ref=e76]

  - cell "21" [ref=e81]
  - cell "  " [ref=e82]

  - cell "23" [ref=e85]
  - cell "  " [ref=e86]
  - cell "24" [ref=e87]
  - cell "1. **Explore project context** — check files, docs, recent commits" [ref=e88]

  - cell "3. **Ask clarifying questions** — one at a time, understand purpose/constraints/success criteria" [ref=e92]
  - cell "27" [ref=e93]

  - cell "## The Process" [ref=e182]
  - cell "72" [ref=e183]
  - cell "  " [ref=e184]
  - cell "73" [ref=e185]
  - cell "**Understanding the idea:**" [ref=e186]
  - cell "74" [ref=e187]
  - cell "  " [ref=e188]
  - cell "75" [ref=e189]
  - cell "- Check out the current project state first (files, docs, recent commits)" [ref=e190]
  - cell "76" [ref=e191]
  - cell "- Before asking detailed questions, assess scope: if the request describes multiple independent subsystems (e.g., \"build a platform with chat, file storage, billing, and analytics\"), flag this immediately. Don't spend questions refining details of a project that needs to be decomposed first." [ref=e192]
  - cell "77" [ref=e193]
  - cell "- If the project is too large for a single spec, help the user decompose into sub-projects: what are the independent pieces, how do they relate, what order should they be built? Then brainstorm the first sub-project through the normal design flow. Each sub-project gets its own spec → plan → implementation cycle." [ref=e194]
  - cell "78" [ref=e195]
  - cell "- For appropriately-scoped projects, ask questions one at a time to refine the idea" [ref=e196]
  - cell "79" [ref=e197]
  - cell "- Prefer multiple choice questions when possible, but open-ended is fine too" [ref=e198]
  - cell "80" [ref=e199]
  - cell "- Only one question per message - if a topic needs more exploration, break it into multiple questions" [ref=e200]
  - cell "81" [ref=e201]
  - cell "- Focus on understanding: purpose, constraints, success criteria" [ref=e202]
  - cell "82" [ref=e203]
  - cell "  " [ref=e204]
  - cell "83" [ref=e205]
  - cell "**Exploring approaches:**" [ref=e206]
  - cell "84" [ref=e207]
  - cell "  " [ref=e208]
  - cell "85" [ref=e209]
  - cell "- Propose 2-3 different approaches with trade-offs" [ref=e210]
  - cell "86" [ref=e211]
  - cell "- Present options conversationally with your recommendation and reasoning" [ref=e212]
  - cell "87" [ref=e213]
  - cell "- Lead with your recommended option and explain why" [ref=e214]
  - cell "88" [ref=e215]
  - cell "  " [ref=e216]
  - cell "89" [ref=e217]
  - cell "**Presenting the design:**" [ref=e218]
  - cell "90" [ref=e219]
  - cell "  " [ref=e220]
  - cell "91" [ref=e221]
  - cell "- Once you believe you understand what you're building, present the design" [ref=e222]
  - cell "92" [ref=e223]
  - cell "- Scale each section to its complexity: a few sentences if straightforward, up to 200-300 words if nuanced" [ref=e224]
  - cell "93" [ref=e225]
  - cell "- Ask after each section whether it looks right so far" [ref=e226]
  - cell "94" [ref=e227]
  - cell "- Cover: architecture, components, data flow, error handling, testing" [ref=e228]
  - cell "95" [ref=e229]
  - cell "- Be ready to go back and clarify if something doesn't make sense" [ref=e230]
  - cell "96" [ref=e231]
  - cell "  " [ref=e232]
  - cell "97" [ref=e233]
  - cell "**Design for isolation and clarity:**" [ref=e234]
  - cell "98" [ref=e235]
  - cell "  " [ref=e236]
  - cell "99" [ref=e237]
  - cell "- Break the system into smaller units that each have one clear purpose, communicate through well-defined interfaces, and can be understood and tested independently" [ref=e238]
  - cell "100" [ref=e239]
  - cell "- For each unit, you should be able to answer: what does it do, how do you use it, and what does it depend on?" [ref=e240]
  - cell "101" [ref=e241]
  - cell "- Can someone understand what a unit does without reading its internals? Can you change the internals without breaking consumers? If not, the boundaries need work." [ref=e242]
  - cell "102" [ref=e243]
  - cell "- Smaller, well-bounded units are also easier for you to work with - you reason better about code you can hold in context at once, and your edits are more reliable when files are focused. When a file grows large, that's often a signal that it's doing too much." [ref=e244]
  - cell "103" [ref=e245]
  - cell "  " [ref=e246]
  - cell "104" [ref=e247]
  - cell "**Working in existing codebases:**" [ref=e248]
  - cell "105" [ref=e249]
  - cell "  " [ref=e250]
  - cell "106" [ref=e251]
  - cell "- Explore the current structure before proposing changes. Follow existing patterns." [ref=e252]
  - cell "107" [ref=e253]
  - cell "- Where existing code has problems that affect the work (e.g., a file that's grown too large, unclear boundaries, tangled responsibilities), include targeted improvements as part of the design - the way a good developer improves code they're working in." [ref=e254]
  - cell "108" [ref=e255]
  - cell "- Don't propose unrelated refactoring. Stay focused on what serves the current goal." [ref=e256]
  - cell "109" [ref=e257]
  - cell "  " [ref=e258]
  - cell "110" [ref=e259]
  - cell "## After the Design" [ref=e260]
  - cell "111" [ref=e261]
  - cell "  " [ref=e262]

  - cell "113" [ref=e265]
  - cell "  " [ref=e266]
  - cell "114" [ref=e267]
  - cell "- Write the validated design (spec) to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`" [ref=e268]
  - cell "115" [ref=e269]
  - cell " - (User preferences for spec location override this default)" [ref=e270]
  - cell "116" [ref=e271]
  - cell "- Use elements-of-style:writing-clearly-and-concisely skill if available" [ref=e272]
  - cell "117" [ref=e273]
  - cell "- Commit the design document to git" [ref=e274]
  - cell "118" [ref=e275]
  - cell "  " [ref=e276]
  - cell "119" [ref=e277]
  - cell "**Spec Review Loop:**" [ref=e278]
  - cell "120" [ref=e279]
  - cell "After writing the spec document:" [ref=e280]
  - cell "121" [ref=e281]
  - cell "  " [ref=e282]
  - cell "122" [ref=e283]
  - cell "1. Dispatch spec-document-reviewer subagent (see spec-document-reviewer-prompt.md)" [ref=e284]
  - cell "123" [ref=e285]
  - cell "2. If Issues Found: fix, re-dispatch, repeat until Approved" [ref=e286]
  - cell "124" [ref=e287]
  - cell "3. If loop exceeds 3 iterations, surface to human for guidance" [ref=e288]
  - cell "125" [ref=e289]

  - cell "126" [ref=e291]
  - cell "**User Review Gate:**" [ref=e292]
  - cell "127" [ref=e293]
  - cell "After the spec review loop passes, ask the user to review the written spec before proceeding:" [ref=e294]
  - cell "128" [ref=e295]

  - cell "131" [ref=e301]

  - cell "## Key Principles" [ref=e316]
  - cell "139" [ref=e317]
  - cell "  " [ref=e318]
  - cell "140" [ref=e319]
  - cell "- **One question at a time** - Don't overwhelm with multiple questions" [ref=e320]
  - cell "141" [ref=e321]
  - cell "- **Multiple choice preferred** - Easier to answer than open-ended when possible" [ref=e322]
  - cell "142" [ref=e323]
  - cell "- **YAGNI ruthlessly** - Remove unnecessary features from all designs" [ref=e324]
  - cell "143" [ref=e325]
  - cell "- **Explore alternatives** - Always propose 2-3 approaches before settling" [ref=e326]
  - cell "144" [ref=e327]
  - cell "- **Incremental validation** - Present design, get approval before moving on" [ref=e328]
  - cell "145" [ref=e329]
  - cell "- **Be flexible** - Go back and clarify when something doesn't make sense" [ref=e330]

  - cell "## Visual Companion" [ref=e334]
  - cell "148" [ref=e335]
  - cell "  " [ref=e336]
  - cell "149" [ref=e337]
  - cell "A browser-based companion for showing mockups, diagrams, and visual options during brainstorming. Available as a tool — not a mode. Accepting the companion means it's available for questions that benefit from visual treatment; it does NOT mean every question goes through the browser." [ref=e338]
  - cell "150" [ref=e339]
  - cell "  " [ref=e340]
  - cell "151" [ref=e341]
  - cell "**Offering the companion:** When you anticipate that upcoming questions will involve visual content (mockups, layouts, diagrams), offer it once for consent:" [ref=e342]
  - cell "152" [ref=e343]
  - cell "> \"Some of what we're working on might be easier to explain if I can show it to you in a web browser. I can put together mockups, diagrams, comparisons, and other visuals as we go. This feature is still new and can be token-intensive. Want to try it? (Requires opening a local URL)\"" [ref=e344]
  - cell "153" [ref=e345]
  - cell "  " [ref=e346]
  - cell "154" [ref=e347]
  - cell "**This offer MUST be its own message.** Do not combine it with clarifying questions, context summaries, or any other content. The message should contain ONLY the offer above and nothing else. Wait for the user's response before continuing. If they decline, proceed with text-only brainstorming." [ref=e348]
  - cell "155" [ref=e349]
  - cell "  " [ref=e350]
  - cell "156" [ref=e351]
  - cell "**Per-question decision:** Even after the user accepts, decide FOR EACH QUESTION whether to use the browser or the terminal. The test: **would the user understand this better by seeing it than reading it?**" [ref=e352]
  - cell "157" [ref=e353]
  - cell "  " [ref=e354]
  - cell "158" [ref=e355]
  - cell "- **Use the browser** for content that IS visual — mockups, wireframes, layout comparisons, architecture diagrams, side-by-side visual designs" [ref=e356]
  - cell "159" [ref=e357]
  - cell "- **Use the terminal** for content that is text — requirements questions, conceptual choices, tradeoff lists, A/B/C/D text options, scope decisions" [ref=e358]
  - cell "160" [ref=e359]
  - cell "  " [ref=e360]
  - cell "161" [ref=e361]
  - cell "A question about a UI topic is not automatically a visual question. \"What does personality mean in this context?\" is a conceptual question — use the terminal. \"Which wizard layout works better?\" is a visual question — use the browser." [ref=e362]
