## ~~Change 2~~ — DROPPED (engineering review finding)

**[Architecture concern]** The blur-then-select approach breaks the native OS Cmd+A
text-selection behavior inside the search box — pressing Cmd+A when the user intends
to select all their typed text would instead blur the input and bulk-select all skills.
This violates the `isEditableTarget` guard's invariant and contradicts the app's own
2-step Escape pattern (which preserves focus discipline).

**Resolution**: Drop Change 2 entirely. Issue #228 ("Cmd+A blocked when search box
focused") is resolved via **discoverability**, not behavior change: the ⌘A badge on
"Select all visible" (delivered by Change 1) makes the keyboard path visible without
hijacking native text-select. The UX gap report's Fix Option 2 ("Add a tooltip / badge
showing ⌘A shortcut") is the correct fix.

# Previous Plan: Skill Delete Protection

## Problem

Users can accidentally delete or unlink skills they rely on. There is no per-skill
protection mechanism in the app — once the delete button is clicked and confirmed,
the skill is gone (modulo the Undo toast window). Power users who curate specific
skills want a deterministic guard, not just an Undo escape hatch.

## Proposed Solution

Add a **per-skill protection toggle** (lock icon) to each SkillItem card. When
locked, the Delete and Unlink buttons are hidden entirely; bulk-delete silently
skips protected skills and surfaces a warning count in the confirm dialog.

Protection state is stored in Redux (persisted via `redux-persist` to
`localStorage`, same pattern as `bookmarkSlice`).

---

## Scope

### In scope

1. **`protectSlice.ts`** — new Redux slice mirroring `bookmarkSlice.ts`.
   Actions: `addProtection(skillName)`, `removeProtection(skillName)`.
   Persisted to `localStorage` via existing redux-persist setup.

2. **`skillItemHelpers.ts`** — `getSkillItemVisibility()` gains an `isProtected`
   input. When true: `showDeleteButton = false`, `showUnlinkButton = false`,
   `showAddButton` unchanged (protection blocks destruction, not creation).

3. **`SkillItem.tsx`** — lock icon button alongside the bookmark button.
   Visual position: left of bookmark (bookmark slides further left when lock shown).
   Active state: `Lock` icon (amber accent, always visible).
   Inactive state: `LockOpen` icon (muted, hover-only opacity).

4. **Bulk delete**: `partitionGlobalDeleteTargets` skips protected names and
   returns them as a new `protectedErrors` array. The `BulkConfirmDialog`
   shows "N protected skills will be skipped" when that array is non-empty.

5. **`getCardContentPaddingClass`** — extend to accommodate lock + bookmark + X
   (three stacked 44px buttons = 132px → `pr-36`).

### Out of scope

- Lock file sync to `~/.agents/.skill-lock.json` (that file is CLI-owned).
- Per-agent protection granularity (global protection only, matches delete scope).
- Import/export of protection list.
- Protection for the Uninstall flow in the Marketplace tab (separate surface).

## Decision Audit Trail

### Auto-decided (6-principle rule)

| Decision                 | Chosen                                           | Reason                                                                                                                                                                  |
| ------------------------ | ------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Active lock color        | `text-foreground` (not `text-amber-400`)         | Amber already has 2 roles: broken symlink + bookmark. A 3rd semantic dilutes both. Design review unanimous.                                                             |
| Inactive lock rest state | `opacity-40` always-visible                      | Non-destructive value-add actions show at rest per DESIGN.md visibility-asymmetry rule. Hover-only discovery is below Linear's bar.                                     |
| Tooltip                  | Add `<Tooltip>` on lock button                   | "Protected — cannot be deleted" / "Click to protect" — lock semantics are not self-evident in operational UI.                                                           |
| `showUnlinkButton` gate  | **Remove `!isProtected` from showUnlinkButton**  | Unlink is reversible (agent-scoped only). Blocking it with protection is an over-broad constraint with no analog in top-tier apps. Protection gates global delete only. |
| IPC enforcement          | UI-gating only, no main-process check            | Threat model is local user (fat-finger prevention, not security). Project explicitly endorses UI-only for local FS signals. Adding IPC sync creates dual-truth problem. |
| `protectedErrors` code   | `outcome: 'error', code: 'EPROTECTED'`           | Reuses existing `BulkDeleteItemResult` shape; gives dialog a machine-readable discriminator without widening the union.                                                 |
| All-locked edge case     | Disable Delete CTA + inline message, skip dialog | Launching a confirm dialog when the outcome is guaranteed "nothing happens" is noise. Linear pattern: inline state, no modal.                                           |

### DESIGN.md additions (from design review)

1. **Amber overload prevention** — amber reserved for broken/inaccessible status + bookmark accent only; lock/protection uses `text-foreground`.
2. **Toggleable protection controls** — always-visible `opacity-40` rest, `opacity-70` hover, `opacity-100` active; pair with tooltip.
3. **Bulk destructive dialog with skipped items** — skip count as secondary muted copy; if all items protected, disable CTA + inline message.
